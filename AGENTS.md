# Mely AI — Codex 工作区指令

## 项目简介

Mely AI 是一款**本地优先的桌面 AI 创作应用**（Windows，RTX 30/40 GPU），产品名「角色工作台 Character Workbench」。

### 我们解决什么问题

现有的 AI 图像生成工具（ComfyUI、Midjourney、SD）只管「生成一张图」，不管「这张图里的角色是谁」。创作者每次生成都要从零拼凑 Prompt、手动维护 LoRA 触发词、翻文件夹找历史参数——角色在不同场景下根本无法保持一致。

对于 Vtuber 创作者来说，这意味着：
- 每次直播封面要反复生成 20–50 张才得到一张「像同一个人」的
- 每次换装要重新找画师委托，$300–$800，等 3–6 周
- 角色 LoRA 没有保护，文件泄露就被人免费盗用
- 声音、外貌、生成历史散落在不同工具中，没有统一档案

### 我们的解法

**以「角色」为第一公民**。每个角色是一个独立容器，绑定其全部多模态身份资产：

```
角色 = 视觉 LoRA + 声音指纹 + 外貌参数(DNA) + 生成历史 + 造型版本树
```

用户在角色语境内创作——只需描述场景，系统自动注入角色 DNA + LoRA + 触发词，保持跨场景一致性。

**一句话价值主张：「你的角色，永远是同一个人。」**

### 目标用户

1. **Indie Vtuber 创作者**（核心）— 高频使用，高付费意愿，需要角色跨封面/素材/周边保持一致
2. **IP / 漫画创作者** — 连载角色一致性，私密性要求高
3. **独立游戏开发者** — 角色概念设计快速迭代，版本分支管理
4. **个人隐私创作者** — 完全本地，不想数据上云

### 与现有工具的本质差异

| 维度 | ComfyUI / A1111 | Midjourney | Mely 角色工作台 |
|------|-----------------|-----------|---------------|
| 核心抽象 | 工作流/节点 | 提示词 | **角色** |
| 一致性 | 手动维护 LoRA 触发词 | --cref（有限） | 角色 DNA 自动注入 |
| 跨模态 | 各工具割裂 | 无 | 图像+声音统一档案 |
| 历史管理 | 无（文件夹堆积） | 无 | 版本树+生成溯源 |
| 隐私 | 本地 | 云端 | 完全本地+加密 |
| 训练入口 | Kohya（难） | 无 | 向导式一键训练 |

## 开发约束（必须遵守）

1. **技术栈已锁定** — Tauri + React/TS 前端, Python FastAPI 后端, SQLite 数据库. 不要提议替代方案.
2. **目标硬件** — RTX 3070 8GB VRAM. 所有涉及 GPU 的方案必须标注 VRAM 需求.
3. **单文件优先** — 代码生成优先考虑单文件可运行，适合 vibe coding 工作流.
4. **任务颗粒度** — 对应「1 人 × 1–3 天」的工作量.
5. **用户优先** — 产品讨论从「用户解决了什么问题」出发，不堆功能.
6. **错误信息** — 所有用户可见的错误用中文自然语言，禁止暴露 Python traceback.
7. **语言** — 代码注释/commit message 英文，UI 文本中文.
8. **Commit 格式** — `[M{N}] {简短英文描述}`

## 文件索引

开始任务前，先读对应文件理解上下文：

### 核心参考文档 (`docs/`)
| 文件 | 内容 | 什么时候读 |
|------|------|-----------|
| `PROJECT_CONTEXT.md` | 技术栈、硬件约束、开发计划、风险项、待决策项 | **每个线程必读** |
| `ARCHITECTURE.md` | 系统架构、ADR 决策记录、进程模型、路由结构 | 涉及架构设计时 |
| `DATA_MODEL.md` | SQLite schema、文件系统布局、索引策略 | 涉及数据层时 |
| `USER_JOURNEY.md` | Vtuber 用户 5 个阶段的痛点和机会点 | 讨论产品需求时 |
| `FEATURE_SPEC.md` | v1/v1.5/v2 功能清单、LoRA 训练参数、TTS 引擎策略 | 确认功能范围时 |
| `DECISIONS.md` | 已确认决策 + 待决策项 | 遇到技术/产品决策时 |

### 可交互产品规格 (`specs/visual/`)
| 文件 | 内容 |
|------|------|
| `character_workbench_spec_final_v2.html` | 完整产品规格（数据模型、核心流程、多模态绑定、功能清单、技术架构） |
| `vtuber_user_journey_deep_dive.html` | Vtuber 用户旅程深度分析（5 个阶段、情绪曲线、痛点、工具链） |

> 这两个 HTML 文件是交互式规格文档，包含 tab 切换、数据表格等结构化信息。docs/ 下的 Markdown 文档从中提取了关键内容，但原始 HTML 是更完整的参考源。

### UI 交互原型 (`prototypes/`)
| 文件 | 内容 | 什么时候读 |
|------|------|-----------|
| `mely-unified.jsx` | 合并原型，包含 4 个模块导航与整体信息架构 | **涉及界面、导航、跨页面流程时优先读** |
| `character-library.jsx` | 角色库主页原型 | 做角色列表、首页入口时 |
| `character-detail.jsx` | 角色详情页原型 | 做角色档案、DNA、历史、声音、安全时 |
| `generation-workbench.jsx` | 生成工作台原型 | 做场景输入、Prompt 组装、批量生成时 |
| `lora-training-panel.jsx` | LoRA 训练面板原型 | 做数据集、训练进度、验证时 |

> `prototypes/` 是正式设计参考输入，不是可直接复制到生产代码中的实现。开发时应以其信息架构、状态组织和交互意图为准，用 TypeScript 重新实现。

## 线程使用方式

每个模块开独立线程，启动时告诉 Codex：

```
请先读取:
1. docs/PROJECT_CONTEXT.md（全局上下文）
2. specs/M{N}_{MODULE}.md（当前模块任务）
3. 如果涉及界面或交互，再读 prototypes/mely-unified.jsx 和对应模块原型

然后开始开发。
```
