## 2) 已固定共享约定

### 2.1 训练模式命名（固定）

- `轻量`：面向 8GB 安全余量场景；默认低步数、低 rank。
- `标准`：M1 默认模式；目标适配 RTX 3070 8GB。
- `精细`：需要 12GB VRAM，3070（8GB）默认不可推荐，必须给出明确提示。
- 模式语义保留不变，但 **执行层模型可自动降级**，不把模式名和底层模型名绑死。

> UI 文案只展示：`轻量 / 标准 / 精细`，不再新增别名。

### 2.1.1 默认模型口径（固定）

- `策略默认模型`：`flux-schnell`
- `3070 8GB 执行层默认模型`：`sdxl`
- `flux-dev`：只保留实验选项，不作为默认

> 说明：这是 M1B 回填后的总线固定口径。没有这层区分，M1E 会在“产品默认”与“8GB 可执行默认”之间反复返工。

### 2.2 训练状态命名（固定）

统一训练业务状态（用于后端与前端对齐）：

- `draft`（草稿）
- `queued`（排队中）
- `preparing`（准备中）
- `training`（训练中）
- `sampling`（采样中）
- `validating`（验证中）
- `completed`（已完成）
- `failed`（失败）
- `canceled`（已终止）

> 说明：M0 通用任务队列状态继续使用 `pending/running/completed/failed`；M1 训练状态是更细粒度业务状态，映射到任务队列即可。

### 2.3 采样频率（固定）

- 训练全程按 20% 间隔产出采样图：`20/40/60/80/100`。
- 采样失败不应导致训练主任务直接崩溃；需记录失败并继续训练，同时给出中文提示。

### 2.4 验证图标准视角（固定）

- 正面
- 3/4 侧面
- 背面
- 特写

### 2.5 用户可见错误文案原则（固定）

- 全部中文自然语言，不暴露 traceback。
- 直接说明“发生了什么 + 用户下一步怎么做”。
- 对硬件限制必须直说：例如 `显存不足，请改用轻量或标准模式`。
- 同类错误文案保持一致，不同页面不重复发明说法。

### 2.6 训练下发硬边界（固定）

- `VRAM < 24GB` 时，不直接下发 FLUX 训练作业。
- 3070 8GB 选择 FLUX 时，统一自动降级到 `SDXL`，不做“先试错再降级”。
- `flux-schnell` 若进入训练链路，必须带 schnell training adapter。
- `flux-dev` 若进入训练链路，必须先确认非商用许可风险。

### 2.6.1 无真机时的 mock 推进规则（固定）

- 当前若没有 3070 8GB 真机，允许后续线程统一使用“3070 8GB mock 设备画像”继续推进开发与联调。
- mock 设备画像至少固定包含：
  - `gpu = RTX 3070`
  - `vramGB = 8`
  - `runtimeDefaultModel = sdxl`
  - `strategyDefaultModel = flux-schnell`
  - `vram < 24GB => 不直接下发 FLUX 训练`
  - `自动降级 = 确定性降级，不做试错`
- mock 推进只允许用于：
  - 前后端合同联调
  - 页面状态联调
  - 错误文案联调
  - 采样 / 验证 / 重训流程联调
- mock 推进**不等于**：
  - 真机显存结论
  - 真机速度结论
  - 真机稳定性结论
  - M1B 最终收口
  - M1H 最终验收签字
- 所有基于 mock 的回填都必须明确写出：`本轮为 mock 3070 8GB 联调，不代表真机结果`。
