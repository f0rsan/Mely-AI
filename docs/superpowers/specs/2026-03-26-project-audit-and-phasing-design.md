# Mely AI 项目审查与分阶段推进设计稿

## 1. 目的

这份文档不是功能说明书，而是项目级审查结果与推进设计。目标是先统一对产品本质、当前状态、关键风险和推进顺序的理解，再进入每个模块的具体实施计划。

当前工作区已经有产品文档和一组 React JSX 交互原型，但还没有真正的应用代码，因此此时最需要的不是直接实现某个功能，而是先确定：

- 这个产品真正的核心交付是什么
- 当前项目离那个核心交付还差哪些地基
- 应该按什么顺序推进，才能尽快验证产品价值，同时避免返工

## 2. 产品本质理解

Mely AI 不是通用的 AI 出图工具，也不是把现有模型和开源工具简单包一层桌面壳。它的核心价值在于“角色持续一致性”，也就是把一个角色在视觉、声音、设定、历史和版本变化上的身份稳定下来。

对目标用户来说，真正痛的不是“不会生成图”，而是：

- 每次生成都要重新拼 Prompt 和 LoRA 触发词
- 同一个角色在不同场景里像不同的人
- 换装、周边、封面、语音等资产彼此割裂
- 没有统一档案，也没有本地保护能力

所以 Mely AI 的核心抽象必须始终是“角色”，而不是工作流、模型、节点或单次任务。

如果这个方向被执行对了，用户得到的是：

- 一个可长期复用的角色档案
- 一条无需反复手动维护参数的创作主链路
- 一个完全本地、可保护、可追溯的角色工作台

## 3. v1 真正要证明什么

虽然文档中列了大量功能，但 v1 不是要证明“我们能把很多 AI 能力塞进桌面应用”，而是要证明以下三件事同时成立：

### 3.1 角色可以被稳定建立

用户能通过图片或文字创角进入训练流程，得到一个足够稳定的角色基础模型，并把它保存成可复用的角色档案。

### 3.2 角色可以被稳定复用

用户在生成新封面、新场景图时，不需要反复拼技术参数，系统能自动注入角色信息，让产出保持“像同一个人”。

### 3.3 角色资产可以被稳定管理

训练结果、声音、生成历史、造型分支和导出材料，都回到同一个角色容器，而不是散落在多个目录和工具里。

如果这三件事不能被证明，后面的声音、加密、时间戳和导出都只是附加功能，不构成产品成立。

## 4. 当前项目现状审查

### 4.1 已有资产

当前项目已经具备较完整的文档体系：

- 产品定位
- 技术架构
- 数据模型
- 用户旅程
- 功能规格
- 模块拆解
- 已确认决策与待决策项
- 一组可用于提取交互意图的 UI 原型

这说明产品思路和主要交互方向都已经比较清楚，且目标用户、技术栈、硬件约束和阶段目标都基本锁定。

### 4.2 当前缺口

当前工作区已经有文档和原型，但还没有真正的应用代码，意味着以下能力尚未落地：

- 桌面应用骨架
- Python sidecar 启动链路
- 数据目录与数据库初始化
- 模块之间可复用的任务与状态基础设施
- 下载与模型管理地基
- 任何可执行、可验证的主流程

此外，当前目录也不是一个 git 仓库，这会直接影响后续测试、审查、提交与阶段化推进。

### 4.3 结论

当前项目不是“已有产品，缺几个功能”，而是“产品定义和交互意图比较成熟，但工程实现还未开始”。因此正确推进方式不是横向同时做多个模块，而是先立地基，再逐步打通最短价值链路。

## 5. 关键约束

项目推进必须始终遵守以下硬约束：

- 技术栈固定为 Tauri + React/TypeScript + Python FastAPI + SQLite
- 面向 Windows，本地优先
- 目标硬件为 RTX 3070 8GB VRAM
- 用户可见错误必须是中文自然语言
- 单人 1–3 天可完成的粒度才适合作为实际执行单元

这意味着所有设计都必须偏向：

- 可拆分
- 可验证
- 先主链路、后扩展能力
- 先证明成立、后追求完整

## 6. 项目真正的三大风险

### 6.1 工程地基风险

如果 M0 没把桌面壳、后端生命周期、SQLite、任务基础设施和按需下载打稳，后续所有模块都会在不稳定基础上反复返工。

### 6.2 核心价值风险

如果 M1 和 M2 不能在 RTX 3070 8GB 上证明“角色训练 + 一致性生成”是可用的，那产品最核心的承诺就站不住。

### 6.3 体验断裂风险

如果训练、生成、声音、造型和历史各自成立，但没有被统一回角色容器中，用户得到的仍然是一组拼装工具，而不是角色工作台。

## 7. 正确的推进顺序

### 阶段 0：工程起步整理

这不是 M0 的正式业务任务，而是让项目进入可开发状态的前置阶段。至少包括：

- 建立真正的代码仓库
- 建立基础目录结构
- 确定开发入口和测试入口
- 让后续每个模块都有统一起点

如果这一步跳过，后续所有“计划”和“提交规范”都只停留在纸面。

### 阶段 1：M0 地基层

目标不是一次性做完整个 M0，而是拆成三个递进子阶段：

- M0-A：桌面壳、后端、数据库、健康检查
- M0-B：角色基础数据与文件系统管理
- M0-C：任务队列、下载框架、最小角色库首页

这样拆的原因是：

- 先证明应用能站起来
- 再承接角色数据
- 最后再接高风险基础设施

### 阶段 2：M1 视觉训练

M1 的第一优先级不是 UI，而是模型与显存可行性验证。必须尽快证明：

- FLUX 路线在 3070 8GB 上可运行
- 训练速度在可接受范围内
- 训练失败能被翻译成用户能理解的话

只有这个成立，数据集导入、DNA 配置和进度界面才有意义。

### 阶段 3：M2 生成工作台

M2 是第一个真正直接体现产品价值的阶段。到这一阶段，用户必须能完成：

- 选择角色
- 输入场景
- 自动注入角色信息
- 生成一致图片
- 把结果自动归档回角色历史

这是 v1 最关键的“价值验证线”。

### 阶段 4：M3 声音绑定

声音会显著提升产品差异化，但不应抢在角色一致性主链路之前。否则会出现声音能用、角色不稳的倒置状态。

### 阶段 5：M4 造型、导出与资产保护

这是“从好用走向可沉淀、可商用、可保护”的阶段。适合在创作主链路稳定后推进。

### 阶段 6：M5 打包与用户测试

只有当前面链路跑通，M5 的安装引导、Onboarding 和用户测试才有真实价值。否则测试得到的只会是基础缺失带来的噪音。

## 8. 第一阶段的定义

基于当前工作区状态，最先应该进入详细计划的不是完整 M0，而是：

“项目起步整理 + M0-A 最小地基”

它的目标是：

- 让项目变成真正可开发的代码仓库
- 建立桌面应用、后端和数据库的最小闭环
- 为后续 M0-B 和 M0-C 提供可复用入口

这比直接把 M0 整块展开更合理，因为它既符合单人 1–3 天粒度，也最容易被验证。

## 9. 成功判断标准

项目推进是否走在正确方向上，不看功能数量，而看以下检查点是否被依次通过：

### 检查点 1：工程成立

- 代码仓库建立
- 开发入口可运行
- 测试入口可运行
- 前后端和数据库形成最小闭环

### 检查点 2：角色建立成立

- 用户可以把参考图或文字描述带入角色训练流程
- 训练结果能进入角色档案
- 基础角色信息可被复用

### 检查点 3：角色复用成立

- 用户可以通过一句场景描述生成图
- 输出结果保持角色一致性
- 生成历史和参数回到角色容器中

### 检查点 4：角色资产管理成立

- 造型、声音、导出、加密和证明围绕角色统一工作

只要检查点 1 到 3 不成立，就不能说产品的核心承诺已经站住。

## 10. 本文结论

Mely AI 的正确推进策略不是“按文档顺序把功能全部做出来”，而是：

- 先把工程和数据地基打稳
- 尽早验证 3070 8GB 下的训练与一致性生成
- 始终围绕“角色容器”而不是单点功能扩张

因此，下一份真正需要写的执行计划，应该聚焦在：

“项目起步整理 + M0-A 最小地基”

而不是直接展开整个 M0 到 M5。
