# M0 基础骨架设计稿

## 1. 背景

当前工作区只有产品文档和模块拆解，还没有真正的应用代码。M0 原始范围包含项目骨架、数据库、角色 API、文件系统、任务队列、按需下载和首页 UI，跨度过大，不适合作为同一轮设计和实现。

本设计稿将 M0 拆成第一个可独立交付的子项目，只解决“应用能不能站起来”的问题，为后续角色管理、任务调度和模型下载提供稳定地基。

## 2. 本轮目标

本轮只交付一个最小可运行骨架，满足以下结果：

- 桌面应用可以在开发模式启动。
- 前端页面可以加载，并主动检查后端状态。
- Python 后端会随桌面应用启动。
- 后端会自动创建本地数据目录和 SQLite 数据库。
- 后端会自动执行数据库初始化和版本迁移。
- 用户在首页能看到明确的中文状态提示。

## 3. 本轮不做

以下内容明确不在这一轮范围内：

- 角色创建、编辑、删除接口
- 文件上传和角色目录细分管理
- 任务队列和进度推送
- 按需下载框架和断点续传
- ComfyUI、AI-Toolkit、TTS 引擎接入
- 训练、生成、声音、导出等业务流程
- 完整角色库首页，只保留最小状态页
- Windows 发布打包和 Python 运行时分发

这轮的验收标准是“开发模式地基完成”，不是“可发布安装包完成”。

## 4. 设计原则

- 只做后续模块都会复用的底层能力。
- 启动链路要尽量短，先保证“能跑通”，再叠加功能。
- 前后端契约要从第一天开始固定，避免后续模块各自定义。
- 数据目录和数据库初始化必须可测试，不能依赖人工手动准备。
- 所有用户可见错误都使用中文自然语言，不暴露技术堆栈。

## 5. 总体方案

### 5.1 前端

前端使用 Tauri 2.x + React + TypeScript + Vite。当前阶段只保留一个最小首页，不引入复杂业务状态管理。

首页承担三件事：

- 应用启动后调用后端健康检查接口
- 显示后端连接状态、数据库状态和数据目录路径
- 在失败时显示中文错误信息，并提供重试按钮

首页不是产品正式界面，而是“系统是否已经站稳”的状态页。

### 5.2 后端

后端使用 Python FastAPI，负责：

- 提供 `/api/health` 健康检查接口
- 启动时创建本地目录
- 启动时打开或创建 SQLite 数据库
- 启动时执行数据库迁移
- 对前端返回结构化状态信息

后端内部从第一轮开始拆成清晰职责，而不是把所有逻辑堆在单文件里。至少要区分：

- 应用入口
- 配置与数据目录解析
- 数据库连接
- 迁移执行器
- 健康检查接口

### 5.3 桌面壳

桌面层只负责两件事：

- 启动前端窗口
- 在开发模式拉起 Python 后端，并在应用关闭时结束该进程

这一轮只要求开发模式成立。生产环境如何随安装包分发 Python 运行时，不在本轮解决，但后端入口形式要为后续打包预留稳定接口。

## 6. 目录与边界

建议目录结构如下：

```text
src/                 前端页面与应用入口
src-tauri/           Tauri 配置与桌面壳
backend/
  app/               FastAPI 应用代码
  migrations/        版本化 SQL 脚本
  tests/             后端测试
```

数据目录使用 `~/.mely/`，并在测试环境支持覆盖，避免污染真实用户数据。

初始化时创建以下顶层目录：

- `~/.mely/db/`
- `~/.mely/characters/`
- `~/.mely/models/`
- `~/.mely/temp/`

虽然本轮不会真正写入角色和模型内容，但先把顶层目录立住，可以避免后续每个模块重复发明初始化逻辑。

## 7. 数据库方案

### 7.1 初始化策略

数据库文件路径固定在 `~/.mely/db/mely.db`。后端启动时按以下顺序执行：

1. 解析数据根目录
2. 创建缺失目录
3. 打开或创建 SQLite 文件
4. 确保存在迁移记录表
5. 按顺序执行未应用的 SQL 迁移

### 7.2 迁移策略

采用最简单、可读、可审计的迁移方式：

- 每个迁移是一个按版本排序的 SQL 文件
- 数据库中维护迁移记录表
- 启动时只执行尚未应用的迁移
- 已执行迁移不可重复执行

这样做的原因是：

- 当前项目处于早期阶段，不需要额外引入复杂迁移框架
- SQL 文件最直观，便于后续人工审查和版本管理
- 后面新增表或索引时可以直接追加迁移，不需要重写初始化逻辑

### 7.3 本轮建表范围

本轮直接根据 [docs/DATA_MODEL.md](/Users/yangyingjia/Mely%20AI/docs/DATA_MODEL.md) 建立核心表和索引，原因是这些表属于稳定地基，后续模块都会依赖。

本轮会创建：

- `characters`
- `character_dna`
- `visual_assets`
- `voice_assets`
- `costumes`
- `costume_previews`
- `generations`
- `generation_tags`
- 对应索引
- 迁移记录表

本轮不会为这些表提供业务接口，只保证表结构存在、可初始化、可测试。

## 8. 前后端数据流

启动链路如下：

1. 用户启动桌面应用
2. Tauri 拉起前端窗口，并启动 Python 后端
3. FastAPI 启动后完成目录初始化、数据库连接和迁移执行
4. 前端加载首页后调用 `/api/health`
5. 后端返回：
   - 服务是否在线
   - 数据目录路径
   - 数据库文件路径
   - 迁移是否成功
6. 前端根据结果展示中文状态

健康检查返回的数据既服务于当前状态页，也为后续设置页和诊断页提供同一份底层信息。

## 9. 错误处理

本轮错误处理只处理“用户会在启动阶段遇到的问题”，包括：

- 后端未启动
- 数据目录创建失败
- 数据库打开失败
- 迁移执行失败

处理原则：

- 前端只展示中文自然语言，例如“后端未启动，请重试”“数据库初始化失败，请稍后再试”
- 后端日志保留技术细节，便于开发排查
- 健康检查接口在失败时返回可判断的状态，而不是让前端猜测

## 10. 测试策略

### 10.1 后端测试

后端测试是这一轮的核心验证手段，至少覆盖：

- 临时数据目录下的目录初始化
- 空数据库首次启动后的建库与建表
- 重复启动时迁移不会重复执行
- `/api/health` 返回预期结构
- 迁移失败时返回可识别错误

### 10.2 前端测试

前端只做最小测试，覆盖：

- 正常状态渲染
- 健康检查失败时显示中文错误
- 点击重试后会重新请求

### 10.3 人工验证

本轮在开发模式做三项人工验证：

- 桌面窗口能打开
- 首页能显示后端正常状态
- 本地数据目录和数据库文件确实被创建

## 11. 验收标准

本轮完成的判断标准如下：

- `tauri dev` 能启动桌面窗口
- Python 后端会随应用启动
- 前端能成功访问 `/api/health`
- `~/.mely/` 目录会自动创建
- SQLite 数据库文件会自动创建
- 所有基础表和索引会自动建立
- 后端测试通过
- 用户可见错误均为中文自然语言

## 12. 风险与控制

### 风险 1：开发环境下的 Python 启动方式不稳定

控制方式：

- 本轮只针对开发模式打通
- 后端入口保持单一且固定，后续打包阶段再处理运行时分发

### 风险 2：一次性建立全部核心表，后续调整成本较高

控制方式：

- 当前表结构来自已确认的数据模型文档，属于稳定基础
- 后续变更通过追加迁移处理，而不是修改历史迁移

### 风险 3：首页过早产品化

控制方式：

- 明确把首页定义为“系统状态页”，不是正式角色库页面
- UI 只承担诊断和确认作用，不承载产品叙事

## 13. 本轮之后的自然衔接

这一轮完成后，后续模块可以直接在现有地基上继续：

- 下一步可接角色 CRUD 和角色目录管理
- 再下一步可接任务队列和下载框架
- 前端首页也能从状态页自然扩展成角色库首页

## 14. 实施边界结论

这轮 M0 子项目定义为：

“先做一个可启动、可连通、可建库、可诊断的最小桌面应用地基。”

只要这个地基成立，后续所有模块都能在同一套目录、数据库和前后端契约上继续生长，而不需要返工启动链路。
