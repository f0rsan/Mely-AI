# M2 生成工作台 Bus 总线设计稿

## 1. 目标

这份文档不是某一个功能点的实现说明，而是 M2 阶段的总线设计稿。它解决的问题是：

- M2 应该拆成哪些可以独立推进的小任务
- 这些任务之间的依赖顺序是什么
- 哪些内容由 bus 线程统一管理，哪些内容交给叶子任务线程
- 每个任务做完以后，应该怎么回到 bus 线程收口

M2 的核心目标不是“做几个生成相关页面”，而是打通第一个真正给用户交付价值的主链路：

1. 选择角色
2. 输入场景描述
3. 自动注入角色信息
4. 生成一致性图片
5. 自动归档到角色历史
6. 可以再次查看和复用

只要这条主链路没有成立，M2 就不能算完成。

## 2. 当前起点

基于当前仓库审查，M2 不是从零开始，而是建立在已经可运行的 M0 和 M1 地基之上：

- 前端、后端、Tauri 壳和 SQLite 已经存在
- 角色列表、任务队列、下载框架、基础角色数据读写已具备
- 数据模型中已经有 `generations`、`generation_tags`、角色目录和造型相关表
- 当前测试与构建状态正常，说明可以在现有基础上继续拆 M2

M2 设计必须遵守以下约束：

- 技术栈固定为 Tauri + React/TypeScript + FastAPI + SQLite
- 目标硬件是 RTX 3070 8GB VRAM
- 所有用户可见错误必须是中文自然语言
- 每个执行单元必须控制在单人 1–3 天内可收口

## 3. 这次 M2 bus 要解决的旧问题

上一阶段的经验说明，单靠“列任务”是不够的。M2 需要显式修正以下执行问题：

- 没有在每个任务开头统一执行 superpowers 流程
- 没有 bus 线程统一管理任务顺序和依赖
- 任务颗粒度过大，导致一个任务里混了契约、后端、界面、验证多个层次
- 做完以后没有统一收口，导致模块状态分散在多个线程里

因此，M2 bus 线程必须承担流程管理职责，而不是自己吞掉所有实现。

## 4. Bus 线程职责

bus 线程只负责以下内容：

- 定义 M2 的模块边界
- 定义各叶子任务的执行顺序
- 维护依赖门槛和进入条件
- 统一每个任务的验收标准
- 汇总每个叶子任务的完成状态、风险和下一步建议
- 决定何时允许某条叶子线合回 M2 集成线

bus 线程不直接负责：

- 在同一个线程里连续实现多个叶子任务
- 跳过依赖顺序提前做后续功能
- 用一次大提交把多个子模块揉在一起

## 5. 推荐推进方式

### 5.1 推荐方案：契约优先

M2 采用“契约优先”的推进方式，而不是“引擎优先”或“界面优先”。

原因如下：

- 先做引擎运行层，容易让整个模块被 ComfyUI 稳定性拖住
- 先做界面，后续往往会因为状态流和错误流不清楚而返工
- 先把主链路契约定清楚，后面的运行层、界面层、归档层都能围绕同一套输入输出工作

这里的“契约”指的是：

- 生成请求需要哪些输入
- 生成过程有哪些状态
- 成功后产出哪些结果
- 失败时怎样给出用户能理解的提示
- 归档时需要保留哪些参数和文件

### 5.2 不采用的方案

不采用“先把 ComfyUI 跑起来再说”的方案。那样很容易把 M2 变成纯技术接入项目，偏离用户主链路。

不采用“先把生成界面照原型做出来”的方案。那样会产生大量占位逻辑，后面状态流一改，界面就得重做。

## 6. M2 子模块拆解

M2 拆成 8 条叶子任务线，每条线对应一个清晰责任。

### M2-B 生成主链路契约

负责定义整条链路的业务契约：

- 角色选择输入
- 造型选择输入
- 场景描述输入
- 高级参数输入
- 进度状态与结果状态
- 归档数据结构
- 用户可见错误文案

这是后续所有 M2 子模块的对齐基准。

### M2-C 图像引擎运行层

负责图像引擎的生命周期：

- 按需启动
- 健康检查
- 崩溃重启
- 重试上限
- 与训练任务的 GPU 互斥

它只负责“引擎可用”，不负责决定界面长什么样。

### M2-D Prompt 自动组装层

负责把以下信息组装成最终生成输入：

- 用户场景描述
- 角色 DNA Prompt
- LoRA 触发词
- 造型附加 Prompt
- 可选高级参数覆盖

同时定义优先级和冲突规则，保证用户看到的完整 Prompt 与系统实际发送的一致。

### M2-E 单张生成主界面

这是 M2 第一条真正交付给用户的可见价值链路，范围只包括：

- 选角色
- 选造型
- 输入场景
- 查看完整 Prompt
- 发起生成
- 查看进度
- 预览结果
- “再来一张”重新生成

这条线先不承担批量任务和历史图库职责。

### M2-F 生成结果归档

负责把单张生成结果稳定落回角色容器：

- 图片文件保存到角色目录
- 参数快照写入数据库
- 用途标签写入数据库
- 返回可供历史列表展示的数据

M2 的“产品成立感”很大一部分取决于这里，因为用户需要感受到这不是临时生成器，而是角色资产会被自动沉淀。

### M2-G 批量生成队列

负责把多场景输入接到现有任务队列上：

- 多场景提交
- 排队状态
- 顺序执行
- 进度展示
- 完成通知

它建立在单张生成和归档已经成立的前提上，不应反向驱动前面的设计。

### M2-H 历史图库

负责让用户真正“找回以前做过的图”：

- 缩略图浏览
- 按造型筛选
- 按标签筛选
- 按日期筛选
- 查看参数详情
- 从历史参数重新生成

这是归档能力对用户可见的出口。

### M2-I 备用模式与稳定性收尾

负责当主引擎不稳定时仍然保证主链路不中断：

- 连续崩溃计数
- 达到阈值后进入备用模式
- 清楚告诉用户当前引擎已切换
- 保证最基础的文生图仍可继续

这条线只在前面主链路已打通后进入，避免过早扩散范围。

## 7. 推荐执行顺序

推荐的顺序如下：

1. M2-B 生成主链路契约
2. M2-C 图像引擎运行层
3. M2-D Prompt 自动组装层
4. M2-E 单张生成主界面
5. M2-F 生成结果归档
6. M2-G 批量生成队列
7. M2-H 历史图库
8. M2-I 备用模式与稳定性收尾

顺序理由：

- 先把契约定清楚，后续都不容易漂
- 先打通单张主链路，再向批量与历史扩展
- 稳定性补强放到主链路成立之后，避免把备用方案做成主任务

## 8. 分支与工作树建议

M2 使用一条模块集成线和若干叶子线。

模块集成线：

- `codex/m2a-generation`

叶子线建议命名：

- `codex/m2b-generation-contract`
- `codex/m2c-engine-runtime`
- `codex/m2d-prompt-assembler`
- `codex/m2e-single-generate-ui`
- `codex/m2f-generation-archive`
- `codex/m2g-batch-queue`
- `codex/m2h-gallery-history`
- `codex/m2i-fallback-hardening`
- `codex/m2j-end-to-end-stabilization`

每条叶子线都应该：

- 只对应一个清晰责任
- 只改与该责任直接相关的文件
- 先合回 `codex/m2a-generation`
- 不直接合入 `main`

## 9. 每条叶子任务的统一流程

每条叶子任务都必须遵守同一套流程，避免执行质量漂移。

### 9.1 开始前

- 先读 `docs/PROJECT_CONTEXT.md`
- 先读本任务对应 spec 和相关原型
- 先确认自己只处理一个叶子范围

### 9.2 技能流程

每个叶子任务必须显式遵守以下顺序：

1. `using-superpowers`
2. 如果任务涉及行为设计或交互变化，先走 `brainstorming`
3. 写计划后再进入实现
4. 实现过程中遵守测试优先
5. 出现异常时走系统化排查
6. 完成前必须做验证
7. 返回前做一次代码审查视角检查

### 9.3 结束时必须回 bus 线程汇报

每条叶子任务结束后，必须向 bus 线程提交统一格式的收口信息：

- 用户现在多了什么能力
- 实际做了哪些验证
- 当前还剩什么风险
- 哪个叶子任务现在可以接着开始

## 10. 验收门槛

### 10.1 单张主链路成立门槛

以下能力全部成立，才算单张生成成立：

- 能选择角色
- 能输入场景描述
- 系统能自动注入角色信息
- 能查看完整 Prompt
- 能看到实时进度
- 能拿到结果图
- 能把结果自动归档到角色历史

### 10.2 批量能力成立门槛

- 能输入多条场景
- 能排队执行
- 能展示等待中、进行中、已完成
- 全部完成后有通知

### 10.3 历史能力成立门槛

- 能按角色查看历史图
- 能按标签、造型、日期筛选
- 能查看某次生成的参数详情
- 能基于旧参数再次生成

### 10.4 M2 整体成立门槛

M2 只有在以下全部成立时才允许收口：

- 单张生成主链路可用
- 批量生成可用
- 自动归档可用
- 历史浏览可用
- 引擎崩溃后可恢复或切换备用模式

## 11. 风险与缓解

### 风险 1：ComfyUI 稳定性拖慢整个模块

缓解方式：

- 把运行层单独拆成叶子任务
- 在 bus 线程上要求先打通单张主链路的契约和界面接口
- 备用模式放在后段补强，不让全模块早期被拖死

### 风险 2：Prompt 规则散落在前后端不同位置

缓解方式：

- Prompt 组装层单独成线
- 定义统一的合并规则和预览输出
- 用户能看到的 Prompt 与实际发送内容必须来自同一逻辑来源

### 风险 3：生成成功但没有资产沉淀感

缓解方式：

- 归档单独成线
- 单张生成打通后立刻接归档
- 历史图库建立在真实归档数据之上，不做假列表

### 风险 4：批量功能反向把单张链路搞复杂

缓解方式：

- 单张生成先成立
- 批量队列只复用已成立的单张生成与归档能力
- 不允许在批量任务里再发明第二套状态模型

## 12. 非目标

以下内容不属于当前 M2 bus 的范围：

- 声音绑定与 TTS
- 造型树完整管理能力
- 设定书导出
- LoRA 加密
- 多用户配置文件

这些能力后续仍然重要，但不应该进入 M2 主链路设计。

## 13. 本文结论

M2 这次不能再按“做几个功能点”的方式推进，而要按“总线 + 叶子任务”的方式推进。

bus 线程负责统一方向、顺序、门槛和收口；叶子任务负责在小范围内做清楚、做验证、做回报。

真正的 M2 第一优先级不是先把某个技术接上，而是先把“角色场景生成主链路”变成一个清晰、稳定、可逐步落地的系统。只要这条链路被 bus 线程管住，后面的单张、批量、历史和备用模式就能有序推进，而不会再次散掉。
